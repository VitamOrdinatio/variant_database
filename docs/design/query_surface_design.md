# Query Surface Design

## Purpose

The query-surface layer defines how persisted evidence becomes consumable.

VDB persists evidence according to semantic persistence doctrine.

Consumers rarely interact directly with raw persistence structures.

Instead, consumers interact through query surfaces.

Query surfaces provide stable, reusable, provenance-preserving views of persisted evidence.

The purpose of this document is to define how query surfaces should be constructed, governed, and exposed.

---

## Design Principles

### Evidence First

Query surfaces should expose evidence.

They should not perform repository-specific reasoning.

Reasoning belongs to downstream consumers.

---

### Persistence and Consumption Are Distinct

Persistence domains organize evidence.

Query surfaces organize access.

These responsibilities should remain separate.

---

### Preserve Identity Spaces

Query surfaces should not collapse identity spaces.

Variant-space evidence remains variant-space evidence.

Gene-space evidence remains gene-space evidence.

Phenotype-space evidence remains phenotype-space evidence.

Bridging may occur, but identity-space boundaries should remain visible.

---

### Preserve Provenance

Every query surface should support provenance reconstruction.

Consumers should be able to determine:

* source repository
* source package
* source artifact
* transport identity
* namespace-resolution history
* persistence lineage

---

### Preserve Future Interpretability

Query surfaces should expose evidence in a manner that supports future questions not anticipated during initial ingestion.

---

## Architectural Position

Query surfaces occupy the final layer of the VDB evidence lifecycle.

```text
TEP Transport
        ↓

Discovery
        ↓

Validation
        ↓

Namespace Brokerage
        ↓

Semantic Persistence
        ↓

Query Surface Generation
        ↓

Consumer Systems
```

Examples of consumer systems include:

```text
RDGP

future RSP consumers

future analytics systems

future reporting systems

future discovery systems
```

---

## Query Surface Philosophy

VDB does not persist evidence for storage alone.

VDB persists evidence to support future discovery.

Query surfaces therefore represent the primary mechanism through which stored evidence becomes useful.

A persistence domain answers:

```text
Where should evidence live?
```

A query surface answers:

```text
How should evidence be consumed?
```

These questions are related but distinct.

---

## Evidence-State Retrieval

A VDB query surface should be understood as an evidence-state retrieval layer.

Most databases expose rows.

VDB exposes evidence states.

A query surface may physically return tabular rows, JSON objects, or materialized records, but those returned structures should represent preserved evidence states rather than isolated database rows.

Examples:

```text
sample-gene evidence state

variant observation evidence state

phenotype-gene semantic prior state

overlay attachment state

provenance reconstruction state
```

This distinction is important because VDB preserves evidence lifecycles.

A query surface should therefore expose not only current values, but also enough identity, provenance, state, and transition context to reconstruct how the returned evidence state came into existence.

Query surfaces may simplify access.

Query surfaces must not erase evidence-state lineage.

---

## Query Surface Categories

VDB should support multiple query-surface classes.

Examples include:

```text
Evidence Surfaces

Overlay Surfaces

Aggregation Surfaces

Discovery Surfaces

Provenance Surfaces

Operational Surfaces
```

Each class serves different consumers.

---

## Evidence Surfaces

Evidence surfaces expose persisted biological evidence.

Examples:

```text
variant evidence

gene evidence

transcript evidence

phenotype evidence

annotation evidence
```

Evidence surfaces should preserve semantic richness.

Evidence should not be reduced prematurely.

---

## Overlay Surfaces

Overlay surfaces expose attachable semantic priors.

Examples:

```text
GSC phenotype-scoped overlays

future pathway overlays

future ontology overlays
```

Overlay surfaces should preserve:

* release identity
* phenotype scope
* semantic channels
* overlay provenance
* overlay version history

Overlay surfaces should remain attachable rather than merged destructively.

---

## Aggregation Surfaces

Aggregation surfaces expose evidence transformed into consumer-oriented forms.

Examples:

```text
variant-to-gene aggregation

sample-gene evidence

gene-level evidence bundles

candidate-review surfaces
```

Aggregation surfaces do not replace underlying evidence.

They are derived representations.

---

## Discovery Surfaces

Discovery surfaces expose information generated by the discovery engine.

Examples:

```text
artifact profiles

schema profiles

identity inventories

ingestion summaries

domain classifications

routing summaries
```

These surfaces support operational understanding of VDB contents.

---

## Provenance Surfaces

Provenance surfaces expose evidence lineage.

Examples:

```text
source package lineage

TEP lineage

namespace resolution history

overlay attachment history

ingestion history

discovery history
```

Every major VDB object should remain provenance-accessible.

---

## Operational Surfaces

Operational surfaces support governance and observability.

Examples:

```text
unresolved identities

ambiguous mappings

validation failures

routing exceptions

orphaned overlays

stale releases
```

These surfaces assist maintenance and auditing.

---

## Identity-Space Preservation

Query surfaces must preserve identity-space boundaries.

Examples:

```text
(sample_id, gene_id)

≠

(phenotype, gene_id)

≠

(variant_id)

≠

(transcript_id)
```

Consumers may bridge these spaces.

Query surfaces should not silently collapse them.

---

## Variant-Centric Surfaces

Variant-oriented surfaces support consumers requiring variant-level detail.

Examples:

```text
variant observations

variant annotations

transcript consequences

frequency annotations

clinical annotations
```

These surfaces are especially important for VAP-derived evidence.

Variant multiplicity should remain visible.

---

## Gene-Centric Surfaces

Gene-oriented surfaces support aggregation-oriented consumers.

Examples:

```text
gene evidence summaries

sample-gene evidence

gene review surfaces

gene candidate surfaces
```

These surfaces are especially important for RDGP.

Gene-centric views should remain reproducible from underlying variant evidence.

---

## Phenotype-Centric Surfaces

Phenotype-oriented surfaces expose semantic priors and knowledge-layer evidence.

Examples:

```text
GSC phenotype overlays

semantic evidence channels

release-scoped phenotype-gene evidence
```

Phenotype context should remain explicit.

Phenotype evidence should not be mistaken for sample evidence.

---

## Overlay Attachment Surfaces

A specialized query-surface category should support overlay attachment.

Examples:

```text
(sample_id, gene_id)
        +
(phenotype, canonical_gene_id)

variant evidence
        +
semantic overlay evidence
```

Attachment methods should remain explainable.

Join logic should remain reconstructable.

Overlay absence should not imply negative evidence.

---

## RDGP-Oriented Surfaces

RDGP is the primary initial consumer.

RDGP-oriented surfaces should support:

* sample-gene aggregation
* provenance reconstruction
* overlay attachment
* uncertainty preservation
* evidence-channel separation
* candidate generation

Query surfaces should support RDGP without embedding RDGP reasoning logic.

---

## Future RSP-Oriented Surfaces

Future RSP integration should be anticipated.

Potential examples include:

```text
gene-expression overlays

differential-expression surfaces

pathway evidence surfaces

cross-modal evidence surfaces
```

The query architecture should remain extensible.

---

## Canonical Identity Usage

Query surfaces should consume namespace-brokerage outputs.

Examples:

```text
canonical_gene_id

canonical_variant_id

canonical_transcript_id

canonical_phenotype_id
```

However, source identifiers should remain recoverable.

Canonical identities support interoperability.

They do not replace source identity.

---

## Query Surface Provenance

Every query surface should expose sufficient provenance to reconstruct:

```text
source repository

source package

source artifact

TEP lineage

namespace-resolution lineage

discovery lineage

persistence lineage
```

Query surfaces should never become provenance dead ends.

---

## Materialized Versus Dynamic Surfaces

VDB may support:

```text
materialized surfaces

dynamic surfaces
```

Materialized surfaces prioritize performance.

Dynamic surfaces prioritize flexibility.

The choice is an implementation concern.

The design requirement is reproducibility.

Both forms should produce equivalent results from equivalent inputs.

---

## Query Surface Stability

Consumers should not depend directly upon internal persistence layouts.

Query surfaces provide the stability boundary.

Persistence structures may evolve.

Query surfaces should remain comparatively stable.

This separation protects downstream consumers from storage evolution.

---

## Relationship to Semantic Persistence Domains

Semantic persistence domains answer:

```text
Where does evidence reside?
```

Query surfaces answer:

```text
How should evidence be exposed?
```

Query surfaces are derived from persistence domains.

They are not persistence domains themselves.

---

## Relationship to Namespace Brokerage

Namespace brokerage establishes identity relationships.

Query surfaces consume those relationships.

Query surfaces should not independently perform identity normalization.

Identity authority remains external.

---

## Relationship to Discovery

Discovery characterizes evidence.

Query surfaces expose evidence.

Discovery and query exposure therefore represent opposite ends of the evidence lifecycle.

---

## Design Outputs

The query-surface architecture should support generation of:

* variant evidence surfaces
* gene evidence surfaces
* phenotype evidence surfaces
* overlay attachment surfaces
* aggregation surfaces
* provenance surfaces
* discovery surfaces
* operational governance surfaces

All outputs should remain reproducible from persisted evidence.

---

## Summary

The query-surface layer transforms persisted evidence into consumable evidence.

It provides stable, provenance-preserving, identity-aware access paths for downstream consumers while preserving semantic richness, overlay attachability, namespace governance, and future reinterpretability.

Query surfaces therefore function as the consumption layer of the VDB architecture, sitting between semantic persistence and downstream reasoning systems.
